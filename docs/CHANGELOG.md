- RC5.5B implemented (pending Owner review and integration): Read-Only Target Collision Inspection — an explicitly requested `--inspect-collisions` mode that, only after a successful RC5.5A local preflight, reads only the plan-referenced target rows required to compare the approved Import Plan with the existing project, buildings, units, and price history, then stops. Comparison uses shared persistence-projection helpers (`src/import/persistence-projection.ts`) extracted from `database.ts` so the inspector compares exactly the stable fields the write path would persist — including project foreign keys (`developer_id`, `location_id`) resolved through bounded developer/location dependency reads, deterministic JSONB metadata (order-independent), and `recorded_at` — excluding only genuinely volatile runtime values (`updated_at`, runtime `last_data_review_at`); the extraction changes no write behavior. Findings are classified `absent`, `exact_match`, `update_required`, `duplicate_target_rows`, `identity_conflict`, or `inspection_error`; a partial comparison can never be labeled `exact_match`. Every returned target row is strictly validated before comparison (id, parent identity, natural-key presence, field shapes); a malformed row becomes a blocking `inspection_error` and is never silently dropped to `absent`. The operation set is validated before any read: unsupported entities, not-exactly-one project, duplicate keys, payload/entity mismatch, and count mismatch fail closed with zero adapter calls, and every supported operation yields exactly one finding (`totalInspectedOperations === findings.length === operations.length === operationCounts.operations`). The write path's exact pre-RC5.5B semantics are preserved: the shared unit projection accepts `string | null | undefined` for `building_id` and passes it through verbatim, so an unresolved building code still omits `building_id` (undefined) rather than clearing it (null), and the inspector treats an unresolvable expected building identity as a blocking `identity_conflict` before any field comparison. Beyond the plan-key checks, the operation set validation confirms every operation's natural key equals its planner form (`slug`, `${slug}:${buildingCode}`, `${slug}:${unitNumber}`, and the full price-history key), every per-entity count invariant (`projects`/`buildings`/`units`/`priceHistoryRows`/`operations`), and that no two price-history operations collapse to the same database persistence key (which excludes `sourceRow`). Reads are plan-key-bounded and batched (project by slug; developer/location by slug; buildings/units by resolved `project_id` and planned code batches; price history by resolved unit ids); each logical query requests the exact PostgREST row count and paginates by ascending immutable `id` until the uniquely collected rows equal that count — completeness is proven by the count, never inferred from a short page, so a server-side maximum-row cap, a missing/malformed/shifting count, rows exceeding the count, duplicate ids across pages, or a non-advancing page all fail closed as `incomplete_result`/`pagination_inconsistent` rather than a false `absent`; a zero-row result is accepted only when the exact count proves zero. Malformed target price-history rows are never dropped: a malformed row whose persistence-key fields are establishable blocks the operation with that exact persistence identity (even when a valid row also matches, preventing a false `exact_match`), and a malformed row whose key cannot be established blocks every operation on its parent unit (`persistence_key_unresolvable`), so a malformed row can never decay into `absent`. Developer and location dependencies are strictly validated (exactly one row; non-empty id; non-empty slug equal to the requested slug). Malformed-row diagnostics and read failures are deterministic and sanitized: per-key error codes are deduplicated and sorted so the report is byte-equivalent regardless of target-row return order, and provider/network failures are mapped to stable safe codes (`project_read_failed`, `dependency_read_failed`, `building_read_failed`, `unit_read_failed`, `price_history_read_failed`, `incomplete_result`, `pagination_inconsistent`) so no URL, credential, or raw provider message can enter a report or the logger. The reader is a closure-based select-only adapter that never exposes the Supabase client as a runtime property and only ever issues `from`/`select`/`eq`/`in`/`order`/`range`; it uses `SUPABASE_PUBLISHABLE_KEY` exclusively, never reads or falls back to `SUPABASE_SERVICE_ROLE_KEY`, and fails closed before any client or network creation when the publishable key is missing. `--dry-run` stays client-, read-, and network-free; `--dry-run` combined with `--inspect-collisions` and inspection without target/hash/confirmation fail closed; production stays blocked; staging stays unconfigured; local still requires the approved `forever-local` identity. Every report states `readOnlyConfirmed: true`, `executeEnabled: false`, and `writesPerformed: 0`; a successful inspection never enters write execution. Hermetic tests (including a 405-operation Coralina-shaped fixture: 1 project, 8 buildings, 198 units, 198 price-history operations) cover full-field comparison, dependency resolution, strict validation, operation coverage, bounded/batched queries, the credential boundary, and the read-only proof without any gitignored source document. A real local read-only proving run remains a separate Owner checkpoint after review and merge.
