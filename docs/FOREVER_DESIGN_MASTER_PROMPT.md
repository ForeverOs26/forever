# Forever Design Master Prompt

Version: 1.0

Status: Living Document

## 1. Design Philosophy

Forever is not trying to look luxurious.

Forever should feel:

- Trustworthy
- Calm
- Intelligent
- Transparent
- Premium through clarity

The interface must avoid flashy effects, decorative noise, exaggerated luxury cues, and visual gestures that distract from decision quality.

Forever should feel premium because the information is clear, the hierarchy is disciplined, the spacing is calm, and the typography is confident. Premium comes from restraint, structure, and judgment.

Every visual decision must support the user's ability to understand a property, assess risk, and move toward a more confident decision.

## 2. Emotional Journey

The Forever experience should guide users through the following emotional flow:

```text
Curiosity
    |
Trust
    |
Understanding
    |
Confidence
    |
Action
```

### Curiosity

The user should immediately understand that the project is relevant and worth exploring. The interface should invite attention without exaggeration.

### Trust

The user should see evidence that Forever is careful, structured, and transparent. Trust signals should appear early and be connected to real data.

### Understanding

The user should be able to understand what the project is, who it is for, what makes it strong, and where uncertainty remains.

### Confidence

The interface should help the user compare, interpret, and decide. Confidence comes from explanation, not persuasion.

### Action

The next step should be clear, calm, and contextual. Action should feel like a natural outcome of understanding.

## 3. Visual Language

### Typography Hierarchy

Typography should create a clear reading path. Headings should establish meaning, subheadings should clarify context, and body text should be easy to scan. Display typography should be used sparingly and only where the content deserves emphasis.

### Spacing Philosophy

Spacing should make the interface feel composed and readable. Sections need enough breathing room to support comprehension, while dense information areas should remain organized and efficient.

### Card Philosophy

Cards should represent meaningful units of information, not decoration. A card should exist only when it helps users compare, group, verify, or act on information.

### Document Philosophy

Documents should feel official, structured, and trustworthy. Any document-like surface should prioritize legibility, evidence, timestamps, verification status, and clear ownership of information.

### Information Hierarchy

The most decision-relevant information must appear first. Scores, risks, verification status, pricing context, and buyer fit should be easier to find than secondary details.

### Color Philosophy

Color should support meaning and hierarchy. It should not create emotional manipulation or excessive brand noise. Accent color should guide attention to important evidence, status, and actions.

### Icon Philosophy

Icons should clarify meaning, not decorate the page. Use icons for recognizable concepts such as verification, documents, location, risk, score, timeline, and contact.

### Animation Philosophy

Animation should be minimal, purposeful, and calm. Motion may support orientation, transitions, or feedback, but it must never distract from reading or decision-making.

Information is always more important than decoration.

## 4. Hero Specification

### Purpose

The hero introduces the project and establishes the first layer of trust, relevance, and context.

### Layout

The hero should prioritize project identity, location, project type, status, primary image, and one clear next step. It should feel immersive but not promotional.

### Content Hierarchy

The first visible content should communicate:

- Project name
- Location
- Project type
- Price or starting price context
- Verification or trust signal
- Primary action

### Photography Style

Photography should show the actual project, location, architecture, materials, or lifestyle context. Images should be clear, inspectable, and relevant. Avoid dark, vague, overly cropped, or generic stock-like imagery.

### Primary Actions

The hero should include one primary action and, if needed, one secondary action. Actions should support inquiry, project review, or comparison.

### Trust Signals

Trust signals should be visible but restrained. Verification, inspection, project status, and data freshness matter more than badges without meaning.

### Five-Second Understanding

Within five seconds, users should understand what the project is, where it is, why it may matter, and whether Forever has structured confidence in it.

## 5. Forever Passport Specification

The Forever Passport is an official digital investment passport.

It is not a dashboard.

It is not an analytics widget.

The Passport is the canonical project summary across Forever interfaces. It should feel official, portable, verified, and structured.

### Hierarchy

The Passport should begin with project identity, Forever ID, verdict, and overall score. Secondary information should support verification, buyer fit, risks, and timestamps.

### Sections

Passport sections should be clear and document-like:

- Identity
- Verdict
- Scores
- Best Buyer Profile
- Recommendation Summary
- Risks Summary
- Verification Dates
- Timeline

### Scores

Scores should be readable and interpretable. They should not feel like gamification. The score system should communicate decision quality and evidence, not entertainment.

### Verification

Verification data should be visible, especially last inspection and last price update. Users should understand when the passport was generated and what data supports it.

### Risk Presentation

Risks should be presented calmly and directly. They must never be hidden beneath positive information or styled as less important than they are.

### Timeline

The Passport timeline should show important verification and update events. It should create confidence that the project record is maintained over time.

### Visual Identity

The Passport should feel like an official Forever document: structured, precise, calm, and premium through clarity.

### QR Verification Placeholder

The design should allow a future QR verification area, enabling users to verify the passport, open it on another device, or share it across interfaces.

### Future Digital Certificate Concept

The Passport may evolve into a digital certificate for property intelligence, verification, and decision history. The visual system should leave room for this future without overpromising it now.

## 6. Forever Intelligence Specification

Forever Intelligence should look like a professional investment report.

It should communicate:

- Recommendation
- Overall score
- Evidence
- Strengths
- Risks
- Buyer profile
- Investment horizon
- Confidence indicator
- Explainability

### Recommendation

The recommendation should be prominent but not sales-like. It should be expressed as an evidence-based conclusion.

### Overall Score

The overall score should be easy to find and paired with context. A score without explanation is incomplete.

### Evidence

Evidence should be visible through source data, structured fields, and clear summaries. Users should be able to understand what the conclusion is based on.

### Strengths

Strengths should explain what supports confidence in the project. They should be specific and tied to project facts.

### Risks

Risks should be equally visible. Risk presentation is a trust-building feature.

### Buyer Profile

Buyer profile guidance should clarify who the project is best suited for. It should avoid implying universal fit.

### Investment Horizon

Investment horizon should help users understand timing, hold period, and expectation management.

### Confidence Indicator

Confidence should reflect data coverage and evidence quality. It should never create false certainty.

### Explainability

Every recommendation must be understandable. The user should not need to trust a black box.

## 7. Project Detail Layout

The ideal project detail layout should follow a decision-building sequence:

1. Hero
2. Passport
3. Trust Summary
4. Intelligence Report
5. Investment Analysis
6. Gallery
7. Floor Plans
8. Master Plan
9. Unit Plans
10. Documents
11. Developer
12. Contact

### Why Each Section Exists

Hero removes uncertainty about identity and relevance.

Passport removes uncertainty about the canonical project summary.

Trust Summary removes uncertainty about verification and credibility.

Intelligence Report removes uncertainty about recommendation logic.

Investment Analysis removes uncertainty about financial rationale.

Gallery removes uncertainty about visual reality.

Floor Plans remove uncertainty about layouts.

Master Plan removes uncertainty about site organization.

Unit Plans remove uncertainty about available product types.

Documents remove uncertainty about supporting evidence.

Developer removes uncertainty about delivery responsibility.

Contact removes uncertainty about the next step.

## 8. Mobile Experience

Mobile should prioritize clarity, speed, and reading comfort.

### Priorities

The mobile interface should surface project identity, passport summary, verdict, score, risks, buyer fit, and primary action early.

### Touch Interaction

Touch targets should feel generous and predictable. Interactions should be simple, direct, and forgiving.

### Reading Flow

Mobile reading should be linear and calm. Dense sections should be broken into clear blocks that can be scanned without losing context.

### Spacing

Spacing should prevent crowding while preserving momentum. Mobile should not feel compressed or overly card-heavy.

## 9. Tablet Booth Experience

The same design language must extend to booth tablets.

Tablet Booth Mode should be:

- Touch-first
- Simple
- Immersive
- Fast

The booth experience should support live conversations. It should help a salesperson and buyer look at the same evidence, discuss the same project passport, and move toward a clear next step.

Tablet layouts should use larger touch targets, clearer section transitions, and faster access to project summaries, passport data, risks, and contact capture.

## 10. Project Cards

Project cards should feel premium, structured, and useful.

### Required Information

Each card should include:

- Project name
- Location
- Project type
- Starting price or price context
- Verification or trust signal
- Overall score or passport preview where available
- Primary image
- Primary action

### Hierarchy

Project name, location, and decision signal should be immediately visible. Secondary details should support comparison.

### Trust Indicators

Trust indicators should be meaningful. Verification, score, inspection, or passport availability should be more important than decorative badges.

### Passport Preview

A project card may preview the Passport through Forever ID, verdict, or overall score. This should be concise and official-feeling.

### Primary Action

The primary action should move the user into project understanding, not premature conversion.

## 11. Design Tokens

Design tokens should express principles, not decoration.

### Corner Radius

Corner radius should feel modern and restrained. It should support clarity without making serious information feel playful.

### Spacing Scale

Spacing should create calm hierarchy and predictable grouping. Related information should feel connected; unrelated information should breathe.

### Typography Scale

Typography should distinguish page-level identity, section meaning, data labels, explanations, and actions. The scale should support reading, not spectacle.

### Shadow Philosophy

Shadows should be minimal. Use depth only where it clarifies layering, focus, or hierarchy.

### Border Philosophy

Borders should organize information quietly. They should support document-like clarity and comparison.

### Accent Philosophy

Accent use should be purposeful. Accent should guide attention to verification, evidence, risk, status, or action.

## 12. Accessibility

Forever must be readable, navigable, and understandable.

### Readability

Text should be clear at all breakpoints. Important information should not depend on tiny labels or low-contrast styling.

### Contrast

Contrast must support comfortable reading and clear status recognition.

### Font Sizing

Font sizing should respect information density and device context. Critical data must remain legible.

### Interaction Clarity

Interactive elements must look interactive. Primary and secondary actions should be visually distinct.

### Keyboard Support

Navigation, forms, and interactive elements should support keyboard access and visible focus states.

## 13. Success Criteria

A redesigned Forever interface succeeds only if:

- Users trust it.
- Users understand it.
- Users feel less uncertainty.
- Users know what to do next.
- The interface feels premium because of clarity.

If the interface looks polished but fails to improve decision confidence, it has failed.
